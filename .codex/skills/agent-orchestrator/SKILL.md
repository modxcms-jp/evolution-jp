# Agent Orchestrator

`.agent/agents/` を参照して、必要な担当を選び、作業順序と書き込み範囲を決める。
スキルは手順と判断基準、エージェントは責務境界と成果物を定義する。

## 参照順

1. `AGENTS.md`
2. `.agent/agents/README.md`
3. `.agent/agents/orchestrator.md`
4. 必要な担当エージェント定義
5. 関連する `.codex/skills/*/SKILL.md`
6. 関連する `assets/docs/`

## 実行ルール

1. 作業開始時に、使用スキル、使用エージェント、使わないエージェントと理由を1-3行で宣言する。
2. 単独で完結する作業は無理に分担しない。
3. 複数担当が有効な場合のみ、`explorer` / `planner` / `worker` / `reviewer` / `tester` / `roadmap-curator` から選ぶ。
4. 実装前に、担当ごとの入力、成果物、書き込み範囲を明示する。
5. 並行作業する場合は、編集対象ファイルが重ならないように分ける。
6. 調査担当は原則読み取り専用にする。
7. 実装担当は指定された書き込み範囲だけを編集する。
8. レビュー担当は差分レビューを行い、実装のやり直しは担当しない。
9. DocumentParser、キャッシュ、DB、イベントに影響する場合は、対応する `assets/docs/` を必ず参照する。

## 開始時の宣言

作業開始時に、次の形式で短く宣言する。

```text
agent-orchestrator を入口にします。今回は <理由> のため <agent> を主担当にします。
使用スキル: <skill>。使わないエージェント: <agent> は <理由> のため使いません。
```

単独作業で十分な場合も、分担しない理由を明示する。

```text
agent-orchestrator を入口にします。今回は差分が小さく調査だけで完結するため explorer 単独で進めます。
worker と tester は、実装・検証が必要になった時点で使います。
```

## コマンド

### /orchestrate <タスク>

1. タスクを分類する。
2. 必要なエージェントを選ぶ。
3. 開始時の宣言を行う。
4. 各担当の入力、成果物、書き込み範囲を決める。
5. 実行順序を決める。
6. 必要なスキルを各担当へ割り当てる。
7. 実装または調査へ進む。

### /agent-review-flow

1. `reviewer` を主担当にする。
2. 開始時の宣言を行う。
3. 必要に応じて `explorer` に関連実装と docs 調査を割り当てる。
4. 必要に応じて `tester` に検証コマンドと再現確認を割り当てる。
5. `reviewer` が指摘を重大度順に統合する。

## 自律判断可能

- 担当エージェントの選定
- 読み取り専用の調査割り当て
- レビュー時の `reviewer` 選定
- ロードマップ整合性確認のための `roadmap-curator` 選定

## 要相談

- 複数エージェントで同じファイルを編集する必要がある場合
- ロードマップの優先順位を大きく変える場合
- ExecPlan が必要な規模か判断が割れる場合
